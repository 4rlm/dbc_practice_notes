    Release 0: Practice Branching in Git

    Creating a branch in Git gives you a "copy" of your project to work on. Here, for instance, is an octopus:

    Aw, just look at that little buddy.

    Let's say you wanted to improve this octopus with two new features: a cowboy hat and some heart-shaped glasses.

    You might make a branch for each feature. You'd have your master branch (the default branch in any repository), which was the original octopus. You would create and check out a cowboy_hat branch and heart_glasses branch. You could switch between branches depending on what you felt like working on. Once a given feature was complete, you could merge it back into the original project. 

    Here's a simplified version of that workflow:

    Our diagram is a bit simpler than real life: the branches do not have to be created at the same time, or merged back in at the same time. Each feature is added back into master individually. So if the glasses are finished before the cowboy hat, no problem: those can be merged first, and the cowboy hat can come later -- or not at all, depending on changing hat trends. Branching allows you to discard a feature or experiment that didn't turn out quite like you hoped.

    try the thing

    Let's practice working with branches. Make sure you're reading your terminal output after each command and making observations about what's happening.

    If you don't have a practice repository handy, let's create one (otherwise you can skip this part):

    mkdir practice-repo
cd practice-repo
git init
touch test_file.txt
git add test_file.txt
git commit -m "Add a test file"

    Okay, on to the good stuff. First, we can create a new branch, check it out, and do some imaginary coding in it:

    git branch
git checkout -b feature-branch
git branch
touch feature_file.txt
git status
git add feature_file.txt
git status
git commit -m "Implement vague and imaginary feature"
git status

    We can switch between branches to see that our new "feature" only exists on our feature branch:

    git branch
git checkout master
git branch
ls
git checkout feature-branch
ls

    Merging Files

    Once you have completed and tested the project on a feature branch, you want to merge it back into your master branch. Merging takes the code from a target branch (your feature) and merges it into the current branch (master). As long as there is no conflict between the code on the two branches this happens seamlessly and easily. Without merge, we would have to copy and paste our new code into our existing code, which could get really complicated if we have changed several files. Luckily, Git does this for us. Yeah Git!

    try the thing

    Now that you're all finished with your pretend "feature," you can merge it back into master.

    git checkout master
git merge feature-branch
ls

    Release 1: Build an Octopus

    In this release, you'll use Git workflow and branches in a project. You can use our Git Workflow guide if you need a quick reference.To see a list of all Git commands, type git help in your terminal. To learn more about a particular command, type git help COMMAND-NAME. 

    The goal of the project is to complete the octopus project shown above. Our octopus starts without anything to wear. Use feature branches to create a hat and a pair of glasses for it. When these features are merged back to your master branch, you'll have an octopus who is ready to go out on the town!

    do the thing
    1. Create an octopus repository, and add an octopus.md file that contains the name and age of your octopus. From here, make sure to do all of your work on feature branches. (It's generally considered best practice to stay off the master branch while working, to avoid breaking your codebase. )
    2. Add a cowboy hat feature by using the proper Git workflow. The feature itself can just be additional information in the octopus.md file, noting that the octopus is wearing a cowboy hat.
    3. Merge your feature into your master branch.
    4. Add a heart-shaped glasses feature using the same proper Git workflow.

    Submit this assignment by uploading the relevant portion of your terminal history (as a screenshot or text file) and a zipped copy of your repo. Thank you for doing your part to make octopi more fashionable!
